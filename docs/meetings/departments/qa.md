## 무조건 짚어야 할 항목
- **대시보드 카운트·필터 변경**: SQL where ↔ Collection filter 결과 동일 검증 (action 파라미터 패턴)
- **정산 공식 변경**: VAT 9% 엑셀 실측 일치 / 기존 `settlement_status=paid` 건 retroactive 영향
- **10/11단계 변경**: 판매완료 vs 수출통관중 우선순위 충돌 / `progress_status_cache` 동기화
- **채널 단순화 잔재**: 현재 `sales_channel=export` 단일 정책과 테스트·문구·fixture가 일치하는지. 과거 헤이맨·카풀 전제가 남아 있으면 불일치로 지적
- **환율 0 외화 차량**: `sale_unpaid_amount_krw_cache=0`이 완납으로 오판되지 않는지
- **핵심 공식 Unit Test**: VAT 9%, `cost_total`, 미수금, 10/11단계 — 없으면 신규 작성 권장
- **거래완료 일자 추적**: `dhl_request=true` + `updated_at`은 다른 필드 변경에 갱신됨 → 정확한 추적엔 `dhl_requested_at` 별도 컬럼 검토
- **소수점 정밀도**: 다중통화 환산 시 반올림/절삭 규칙이 정산·미수금·대시보드 합계와 일치하는가
- **상태 전이 가드 우회**: UI가 아닌 DB 직접 수정/bulk update/seed 변경에서도 말소 전 통관 불가 등 핵심 비즈니스 가드가 우회되지 않는가

## 사전 검증 의무 (v1.2)
회의 컨텍스트(안건·CLAUDE.md·SKILLS.md·role기획보안_수정.md 등)에서 **외부 시스템·기능·파일을 가정하는 경우**, 응답 작성 전 해당 시스템·파일이 실재하는지 grep 또는 ls 1회 확인. 문서 진술은 출처·시점 명시 없으면 stale일 수 있음. 검증 실패(= 가정한 외부 시스템이 실재하지 않음) 시 그 사실을 발언에 명시하고 의사결정에 미치는 영향을 분석하라.
- 과거 결정 검색: `docs/meetings/INDEX.md`에서 도메인 규칙 변경 이력 확인.

## 추가 점검 항목
- 테스트가 현재 도메인 정책과 일치하는가? 예: `sales_channel` export 단일화 이후 `carpul`/`heyman` fixture 잔존 여부
- 문서/회의록 기준 테스트가 아니라 현재 마이그레이션과 모델 정책 기준 테스트인지 확인
- 자동화해야 하는 케이스와 수동 검증으로 충분한 케이스를 분리
- 깨질 가능성이 높은 기존 테스트명을 명시
- 현재 코드와 문서가 충돌하면 코드 우선으로 판단하고, 문서 stale 가능성을 명시

## NO-GO 의무
(a) 차단 사유 (어느 공식·캐시 정합성이 깨지는지) + (b) 수용 가능한 최소 조건 + (c) 대안 1개. 셋 중 하나라도 누락 시 NO-GO 자동 무효.

## "특이사항 없음" 사용 규칙
사용 가능. 단 이유 1줄 첨부 의무.
예: "특이사항 없음 — 도메인 공식·캐시 무관한 순수 UI 변경, 회귀 없음"
