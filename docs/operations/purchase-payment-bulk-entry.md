# 매입대금 일괄 입력/정리 — 운영 기록 & ssancar 대비 설계

> 작성 2026-06-22. import 차량의 매입 입금이력(PBP) 누락 문제를 정리한 작업 전체 기록과,
> **대물량(ssancar, 수천 건) 대비 구조화 export→기입→import 조치 설계**.
> 관련 메모리: `project_import_purchase_paid`, `project_vehicle_import`, `project_excel_export_import`,
> `project_settlement_data_remediation`.

---

## 1. 배경 — 왜 매입대금이 전부 미지급으로 떴나

- `vehicles:import`(헤이맨 수출차량현황표)는 **Option A**(차량 마스터 + 핵심 재무)만 적재하고
  **매입 입금이력(PurchaseBalancePayment, PBP)은 안 만든다.**
- 그래서 import 차량은 `purchase_price`/`selling_fee`(매입합계)는 있는데 지급기록(PBP)이 0 →
  **매입 미지급 accessor**가 전액을 미지급으로 계산 → 재무처리 대기를 가득 채움.
- 추가로 차량을 UI에서 저장하면 `Vehicle::saved` 훅이 **전액 draft PBP를 자동 생성**(SKILLS #23) →
  재무처리에 "확정 대기"로 뜸 (43머3974·21버4376 사례).
- **정산과는 무관** — 정산 자동생성 훅(`Vehicle::saved`)은 ① `auth()->check()` 필수(artisan 미발동)
  ② 거래완료 "전환" 시점만 ③ 정산 이미 있으면 skip. `refreshCaches()`는 raw update(이벤트 X).
  → 매입 PBP 입력은 정산을 **절대 재생성하지 않는다.** (정산처리가 빈 건 별건: 146건 purge 때문.)

## 2. 데이터 출처 & 판별 로직 (헤이맨 "수출차량현황표.xlsx")

판별 기준 파일 = **"1. 헤이맨 수출차량현황표.xlsx"** (최신본). 6/1 import 원본("0." 파일)보다 신규 차량 포함.

| 컬럼 | 의미 | 용도 |
|---|---|---|
| **D** | 차량번호 | 멱등 키 (매칭) |
| **O** | 송금내역확인 | ERP `purchase_remittance_memo`(메모란). **자유 텍스트** — 계약금/잔금/대금/완료/매도비 혼재 |
| **P / Q** | 구입금액 / 매도비 | 매입합계 = P+Q |
| **S** | 재고/판매 단계 | **매입 완료 판별의 핵심** (아래) |
| **BX** | 정산 매입금액 | = P+Q (167/168 일치). "총액"이지 지급 플래그 아님 |

**핵심 통찰 (jin):**
- **S열이 매입 다운스트림 단계**(`입고`/`선적대기`/`선적완료`/`입금대기`/`완료`)면 → 선적·판매까지
  갔으니 **매입대금은 이미 지급된 것**(O 메모 누락·한글숫자·오타여도). S가 보정자 역할.
- **S=`매입대기` 또는 빈칸 + 계약금만** = 진짜 미지급 (최근 매입분).
- O 단독 파싱은 **자유 텍스트라 불신**: 한글숫자("2억5백44만원"), 오타("14,3000,000"), 누락 빈번.
- BX=0 또는 O="취소" → 취소/제외.

## 3. 만든 도구

### `vehicles:mark-import-purchase-paid {xlsx} [--apply]` (dev, 기본 dry-run)
O 송금내역을 파싱해 **ERP 구조대로** PBP 입력:
- O "계약금/계약" → `type=down`(계약금), "잔금/대금/입금/중도금" → `type=balance`(잔금), "매도비" → `type=selling_fee`
- **금액 파서**: `N만(원)`·콤마숫자·6자리+ 추출(날짜 M/D 제외), **각 금액에 최근접 라벨**(거리순) 부착
  — ⚠️ 초기 버그: 고정범위+계약 우선 매칭이 다음 구절 라벨을 오인(104라2951 잔금↔계약금 swap). 최근접으로 교정.
- **완납**(파싱합계 ≈ 매입합계 ±3% OR S 다운스트림): 계약금/매도비=파싱값, 잔금 = **서버 매입합계 차액** 채워 미지급 0.
- **부분**(S 매입대기/빈 + 계약금만): 계약금만 입력, 잔금 미지급 유지.
- 금액 기준 = **서버 현재 매입합계**(파일은 명단·구조 판별용). 기존 미확정 draft 삭제 후 입력. 멱등(note 마커 `import 매입(구조)`).
- `OVERRIDES` 상수 = 자동 미매칭분 jin 수동 매칭(부분 강제/draft 지정).

### `vehicles:import ... --only=차량번호,...` (dev)
- 신규 옵션. 지정 차량번호만 import → **기존 차량 무수정**(신규 추가 전용).
- ⚠️ 옵션 없는 import는 기존 차량을 **forceFill 갱신** → 매입가/매도비 덮어쓰기 위험. 신규만 넣을 땐 `--only` 필수.
- `--with-payments`는 **쓰지 말 것** — 완료정산(paid) + import 입금을 재생성 → purge한 정산이 다시 생김.

## 5. 교훈 (재발 방지)

1. **파일 ↔ 서버 값 불일치**: 6/1 import 후 서버 값이 파일과 미세하게 다름(예 43머3974 매도비 374,820 vs 파일 370,000 — 차액이 매도비에 흡수됨). 그래서 금액은 **서버 매입합계 기준**으로 0 맞추고, 파일은 명단·구조 판별용으로만 사용.
2. **O 자유 텍스트는 자동화 한계**: 한글숫자·오타·누락. S열 같은 **구조화된 보정 신호**가 없으면 신뢰 불가.
3. **import 갱신 부작용**: 옵션 없는 재import는 기존 차량을 덮어씀. 신규만 넣을 땐 `--only`.
4. **정산 ↔ 매입 분리**: 매입 PBP는 정산을 안 건드림(가드 3중 + raw cache update). 헷갈리지 말 것.
5. **확정 vs 대기**: 불확실분은 draft로 넣어 사람이 검토·승인. 확정은 회계잠금이라 비가역.

## 6. ⭐ ssancar 대비 — 구조화 export→기입→import 조치 (설계)

**문제**: ssancar(대물량)에선 이런 매입대금 정리가 **수천 건**. 헤이맨처럼 O열 자유 텍스트를
사람이 매칭/파싱하는 방식은 **확장 불가**. 자동 파서도 자유 텍스트엔 한계.

**해법 = 자유 텍스트 파싱을 버리고, "보기 쉬운 구조화 Excel을 export → 사용자가 기입 → 구조 그대로 import".**
(예정된 `project_excel_export_import` 기능과 동일 철학.)

### 6-1. Export — `vehicles:export-purchase-template`
미지급(또는 전체) 차량을 1행 1차량으로 **기입용 Excel** 생성. 읽기 쉬운 고정 컬럼:

| 차량번호 | 차명 | 매입일 | 매입가 | 매도비 | **매입합계** | **현재 미지급** | 〔계약금〕 | 〔중도금〕 | 〔잔금〕 | 〔매도비지급〕 | 〔지급일〕 | 〔상태〕 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
| (읽기전용, 시스템 채움) | | | | | | | (빈칸 — 사용자 기입) | | | | | confirmed/draft |

- 앞쪽 = 시스템이 채운 **읽기전용 컨텍스트**(차량 식별 + 매입합계 + 현재 미지급).
- 〔 〕 = **노란 배경 기입칸** (서류 양식 패턴 재사용). 합계 검증 셀(`=계약금+잔금+…` vs 매입합계)로 즉시 오기 확인.
- `상태` 드롭다운: `confirmed`(즉시 확정) / `draft`(재무처리 대기) / `skip`.
- 차량번호는 **멱등 키**라 import 시 정확 매칭(자유 텍스트 매칭 불필요).

### 6-2. Import — `vehicles:import-purchase-payments {xlsx} [--apply]`
기입된 템플릿을 읽어 PBP 생성:
- 행별 차량번호로 매칭 → 〔계약금/중도금/잔금/매도비지급〕을 `type=down/balance/selling_fee` PBP로 생성.
- `상태` = confirmed → `confirmed_at=now`, draft → null. `지급일` → `payment_date`.
- **합계 검증**: Σ기입 ≠ 매입합계면 경고(완납 의도인데 차액 있으면 리포트). 부분지급 허용.
- dry-run 기본 + 멱등(note 마커) + 회계 가드 우회는 artisan 컨텍스트.

### 6-3. 이점
- 자유 텍스트 파싱 0 → **오인/swap/한글숫자/오타 문제 원천 제거.**
- 사용자는 보기 쉬운 표에 숫자만 기입 → 수천 건도 일관 처리.
- export(읽기전용 컨텍스트) + 기입칸 분리 = 사람이 "어느 차에 얼마" 헷갈리지 않음.
- 동일 패턴을 **매입/판매입금/정산** 등 다른 일괄 정리에도 재사용 가능.

### 6-4. 상태
**미구현 (설계만).** ssancar 도입 시점 또는 `project_excel_export_import` 구현 시 함께 착수.
