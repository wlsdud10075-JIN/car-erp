## 모델별 합의 사항 (두 AI 모두 지적)

### C1. 서류 다운로드 권한 미비 [심각 · 즉시 수정]

`VehicleDocumentController::show()`에 권한 체크가 없어 `erp` 미들웨어를 통과하는 일반 사용자(영업 role 포함)가 `/erp/vehicles/{id}/documents/deregistration` URL 직접 입력으로 **주민등록번호(nice_reg_owner_rrn)** 가 포함된 말소신청서·양도증명서 PDF를 다운로드 가능합니다.

```php
// 즉시 수정 — VehicleDocumentController::show() 첫 줄에 추가
abort_unless(auth()->user()->isAdmin(), 403, '관리자 권한이 필요합니다.');
```

### C2. N+1 쿼리 — receivableHistories Eager Load 누락 [중요]

`Vehicle::getSaleUnpaidAmountAttribute()` (Vehicle.php:233):
```php
+ $this->receivableHistories->where('method', '!=', 'deposit')->sum('amount');
```
`receivableHistories` 릴레이션이 필요하지만 아래 위치에서 모두 Eager Load 미포함:
- `erp/dashboard` `actionCounts()` → `->with(['finalPayments', 'purchaseBalancePayments'])` 
- `erp/dashboard` `activeVehicles()` → 동일
- `vehicles/index` `vehicles()` computed → 동일

차량 수가 늘수록 N+1 쿼리가 선형 증가합니다.

```php
// 수정: 세 곳 모두 receivableHistories 추가
->with(['finalPayments', 'purchaseBalancePayments', 'receivableHistories'])
```

### C3. actionCounts — clearanceNeeded/shippingNeeded/dhlNeeded 채널 미구분 [중요]

헤이맨·카풀 채널은 수출통관·선적·DHL 흐름이 없음에도 세 카운트 모두 전체 채널을 대상으로 집계됩니다.

```php
// 현재 — 채널 미구분
'clearanceNeeded' => $active->filter(fn ($v) => $v->sale_price > 0 && $v->sale_unpaid_amount <= 0 && ! $v->export_declaration_document)->count(),

// 수정 — export 채널만
'clearanceNeeded' => $active->filter(fn ($v) => $v->sales_channel === 'export' && $v->sale_price > 0 && $v->sale_unpaid_amount <= 0 && ! $v->export_declaration_document)->count(),
```

`applyActionFilter()`의 `clearance_needed` / `shipping_needed` / `dhl_needed` SQL도 동일 수정 필요.

### C4. nice_reg_owner_rrn 평문 저장 [보안 · 중요]

주민(법인)등록번호가 `varchar(20)` 평문 저장입니다. 개인정보보호법상 암호화 저장이 요구됩니다.

```php
// Vehicle.php $casts 추가
'nice_reg_owner_rrn' => 'encrypted',
```

마이그레이션으로 기존 데이터 일괄 암호화 필요.

### C5. 정산 공식 — 헤이맨/카풀 채널 적용 불가 [기능 누락]

`Settlement::getSalesAmountKrwAttribute()`가 `export_declaration_amount × exchange_rate` 기반으로 수출 전용 공식입니다. 헤이맨/카풀 채널에서 정산 생성 시 `export_declaration_amount = NULL` → **마진 0원** 으로 계산됩니다. 채널별 분기 로직 미구현.

## Codex 고유 발견 사항

### X1. 진행상태 우선순위 충돌 — 판매완료 vs 수출통관중 [도메인 버그]

현재 우선순위 평가 순서:
```
7. sale_price > 0 AND 미입금 <= 0 → 판매완료
...
6. export_buyer_id + shipping_date 입력 → 수출통관중  ← 더 낮은 우선순위
```

판매 완납 후 수출통관 정보(바이어+선적일)를 입력한 차량이 **수출통관중** 이 아닌 **판매완료** 로 표시됩니다.

실제 업무 흐름: 판매완납 → 수출통관 신청 → 수출통관중. 현재 로직대로면 수출통관 정보 입력 후에도 여전히 "판매완료"로 남습니다.

**기획 의도 확인 필요.** 수출 채널에서 `export_buyer_id + shipping_date`가 있으면 "수출통관중"이 "판매완료"보다 우선해야 할 가능성이 높습니다:
```php
// 제안: 순서 변경 (6번 → 판매완료 앞으로 이동)
if ($this->export_buyer_id && $this->shipping_date) return '수출통관중';  // 위로 이동
if ($this->sale_price > 0 && $this->sale_unpaid_amount <= 0) return '판매완료';
```

### X2. thisMonthDone — updated_at 기준 신뢰성 문제 [중요]

```php
$thisMonthDone = ..->where('dhl_request', true)
    ->whereBetween('updated_at', [$ms.' 00:00:00', $me.' 23:59:59'])
    ->count();
```

`dhl_request=true`인 차량에서 이달에 **다른 필드**를 수정하면 `updated_at` 갱신 → 이달 거래완료로 재집계됩니다. 정확한 거래완료 일자 추적을 위해 `dhl_requested_at` 컬럼 도입 필요합니다.

```php
// 마이그레이션 추가
$table->timestamp('dhl_requested_at')->nullable();

// Vehicle saving 이벤트에서
if ($vehicle->dhl_request && !$vehicle->getOriginal('dhl_request')) {
    $vehicle->dhl_requested_at = now();
}
```

### X3. sale_unpaid_amount_krw_cache — 0 vs NULL 혼재 [데이터 정합성]

- 2026-05-08 마이그레이션: `default(0)` 생성
- 2026-05-11 마이그레이션: `nullable()` 변경

마이그레이션 사이에 생성된 차량 레코드의 캐시가 0과 NULL 혼재 가능. `php artisan vehicles:rebuild-caches` 미실행 시 `sale_unpaid` 필터에서 `orWhereNull()` 처리로 커버되지만 데이터 신뢰성 저하.

### X4. refreshCaches 무한 루프 취약 구조 [아키텍처 주의]

현재 `refreshCaches()`가 `DB::table()->update()`로 모델 이벤트를 우회해서 무한 루프 없이 동작합니다. 그러나 이 메서드를 `Vehicle::save()`로 실수로 교체하면 즉시 무한 루프 발생합니다.

현재 주석만 있는 상태이므로 더 명확한 경고 필요:
```php
/**
 * ⚠️ IMPORTANT: DB::table()->update()를 반드시 유지할 것.
 * Eloquent save()로 교체 시 ReceivableHistory::saved → syncFinalPayment → FinalPayment::create
 * → FinalPayment::saved → refreshCaches 무한 루프 발생.
 */
public function refreshCaches(): void
```

### X5. 환율 변경 시 KRW 캐시 미갱신 [데이터 정합성]

`exchange_rate` 변경 시 `Vehicle::saving` 이벤트가 캐시를 갱신하지만, 미래에 별도 환율 테이블을 도입하거나 외부에서 환율을 일괄 변경할 경우 `sale_unpaid_amount_krw_cache`가 stale 상태로 남습니다. 현재 단일 테이블 구조에서는 OK이지만 확장 시 주의 필요.

### X6. 채권위험도 통화 불일치 [도메인 검토]

`getReceivableRiskComputedAttribute()`가 `sale_total_amount`(외화)와 `sale_unpaid_amount`(외화) 비율로 계산합니다. 통화 단위가 일관하므로 비율은 정확하지만, **원화 절대값 기준 위험도** (예: USD 미납 $1,000 = 상대적으로 위험할 수도 있음)를 원할 경우 `sale_unpaid_amount_krw_cache` 기준으로 재정의 검토 필요.

### X7. ReceivableHistory deposit 미러링 — 동기화 누락 엣지케이스

수동 DB 조작이나 타이밍 이슈로 `final_payment_id=NULL`인 `method=deposit` 이력이 생길 경우, `finalPayments->sum()` 집계에서 해당 금액이 누락됩니다. 역방향 체크(미연결 deposit 이력 감지) 로직이 없습니다.

## Claude 자체 분석 고유 발견 사항

### L1. clearance_needed — SQL/Collection 불일치 [엣지케이스]

**SQL 필터:**
```sql
WHERE sale_unpaid_amount_krw_cache IS NOT NULL AND sale_unpaid_amount_krw_cache <= 0
```
**Collection 필터:**
```php
$v->sale_unpaid_amount <= 0   // 외화 단위로 판단, KRW 체크 없음
```

환율 미입력 + 외화 차량에서 USD 기준 완납(unpaid=0)이더라도 `sale_unpaid_amount_krw_cache=NULL`이면 SQL은 제외하지만 Collection은 포함합니다. 대시보드 카운트와 클릭 후 목록 건수가 불일치할 수 있습니다.

(실무적으로 완납 시 환율도 입력되므로 드문 케이스이지만 정합성 원칙 위반)

### L2. VehicleDocumentController — 채널 분기 미검증

라우트에서 `type` 파라미터 유효성 검사가 컨트롤러의 `match()` default에서만 처리됩니다. 컨트롤러가 채널별 접근 제한을 검사하지 않아 카풀 차량 ID로 `/documents/invoice` 직접 요청 시 Proforma Invoice(수출 전용)가 생성될 수 있습니다.

```php
// 추가 검증
if (in_array($type, ['invoice', 'sales_contract', 'ro_cipl', 'con_cipl'], true)) {
    abort_unless($vehicle->sales_channel === 'export', 403, '수출 채널 차량만 가능합니다.');
}
```

### L3. Settlement 음수 마진 처리 없음 [UX]

`total_margin < 0` 시 `settlement_amount`와 `actual_payout`이 음수가 됩니다. UI에서 음수 표시 처리가 없어 표시 혼란 가능. 최소한 `max(0, ...)` guard 또는 경고 뱃지 필요.

### L4. AppServiceProvider::boot() 비어있음 — 타임존은 config에서 처리 [확인 완료]

타임존 `Asia/Seoul`이 `config/app.php`에서 설정됨. `role기획보안_수정.md`의 "수정 완료" 항목과 일치. 이슈 없음.

### L5. 사이드바 breadcrumb — 일반사용자 대시보드 명칭 미분기

`sidebar.blade.php`의 breadcrumb에서 `erp.dashboard`는 'ERP 대시보드'로 고정되어 있습니다. role 분기 후에는 일반사용자는 '내 업무 현황' 등으로 다를 수 있습니다.

## Gemini 분석 고유 발견 사항 (이번 세션 신규)

### G1. 우선순위 재조정 권고 — 보안 먼저

Gemini는 현재 개발 우선순위 1번(일반사용자 대시보드)보다 **권한 세분화·주민번호 암호화**를 먼저 처리할 것을 독립적으로 권고함. Claude, Codex와 동일한 결론.

### G2. PDF/Excel 생성 — Queue 비동기화 필요

dompdf는 동기식(Synchronous) 처리. 다수 사용자가 동시에 문서를 출력하면 PHP 프로세스가 점유되어 전체 시스템이 느려짐.
- 해결: Queue Job으로 비동기 생성 → 포워딩 이메일과 동일 패턴 적용 가능

### G3. 정산 로직 Unit Test 필수

`부가세마진 = purchase_price × 0.09` 등 ERP 핵심 공식에 대한 테스트가 없음. 정산 로직이 틀어지면 ERP 신뢰 자체가 무너짐.

```php
// tests/Unit/SettlementMarginTest.php 생성 권장
public function test_vat_margin_is_nine_percent_of_purchase_price(): void
{
    $vehicle = Vehicle::factory()->create(['purchase_price' => 10_000_000]);
    $settlement = Settlement::factory()->for($vehicle)->create();
    $this->assertEquals(900_000, $settlement->vat_margin);
}
```

### G4. bcmath 확장 — 소수점 정밀 계산

PHP 기본 float 연산은 이진 부동소수점 오차 발생 가능. 환율 × 외화 금액 계산에서 원화 1~2원 오차가 쌓일 수 있음.
- 배포 환경 `php.ini`에 `extension=bcmath` 확인 필요

### G5. Livewire Select 박스 — 대용량 드롭다운 성능

바이어/컨사이니가 수천 건이 될 경우 차량 편집 패널의 드롭다운 렌더 성능 저하.
- 해결: Searchable Select 패턴 도입 (검색어 입력 후 제한 조회)
